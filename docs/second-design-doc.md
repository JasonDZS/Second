# Second 产品设计文档

> **人只做决策,经验永不离职。**
>
> Second(中文名建议:接力)是横跨所有协作工具之上的个人智能体层、决策控制面与自托管执行底座。入口用别人的,引擎用现成的,Second 拥有中间那层:每个人一个带授权边界和长期记忆的个人智能体,让任务不再经人中转,人只在关键决策出现,而所有执行、决策和经验沉淀为可审计、可交接、不随离职流失的组织资产。

---

## 1. 为什么叫 Second

Second 是接力棒。它同时压住了产品的三个核心动作:

任务像接力棒一样在人与智能体之间直接传递,而不是每一棒都要经过人手中转;高风险节点上,棒交回人,人做完决策再交还给智能体继续跑;当一个人离开团队,他的个人智能体把全部项目上下文、历史决策和踩坑记录作为一根完整的接力棒,交到继任者手里。

原名 HAGTrace 描述的是机制(Human-Agent-Gate + Trace),Second 描述的是价值(交接与延续)。机制名适合留给协议层(如 Decision Protocol、Trace Schema),产品名应该讲人话。

---

## 2. 问题:人是协作循环里的中转卡点

Codex CLI、Claude Code、Devin 这一代智能体已经能独立完成真实任务,但它们几乎全部是**个人工具**:领导或同事发来一个任务,仍然需要人去找到对应的工作目录、准备上下文、把任务喂给智能体、盯着执行、再把结果转交给下一个人。智能体的生成能力上去了,但组织的瓶颈从"生成"转移到了"协调"——人成了集成层,负责 review、转交、补上下文、传递结果。

这带来三个后果:

第一,**人是卡点**。任务的流转速度取决于人什么时候有空中转,异步智能体的产能被同步的人工中转抵消。

第二,**过程散落**。每个人的智能体会话、决策理由和执行产物分散在各自的电脑和各个商业模型平台上,组织层面没有任何沉淀,无法审计、无法复用、无法优化。

第三,**经验随人流失**。一个核心成员离职,他与智能体协作积累的全部项目知识、偏好、规则和历史决策一并消失,继任者从零开始。

Second 要解决的就是这三件事:让人跳出中转循环、让协作过程沉淀在一处、让经验在人员更替中延续。

---

## 3. 定位:入口别人的,引擎现成的,中间层是我们的

产品由三个明确的"不做"来定义边界:

**不做聊天与协作入口。**任务的进出口就是团队已经每天使用的 Slack、Linear、ClickUp、飞书、GitHub。这些平台已经开放了 agent 接入(Linear 的 agent 是工作区正式成员,Slack 支持 agent 读取对话上下文),与其和它们抢入口,不如寄生其上。用户不需要迁移任何工作习惯,一个人安装即有价值。

**不做智能体执行引擎。**agent loop(规划、工具调用、代码执行、上下文管理)正在被模型厂商以月为单位迭代,已经商品化。Second 复用 Claude Code、Codex CLI 以及 OpenClaw/OpenCode 等开源自托管引擎作为可替换的执行内核,通过 executor adapter 封装差异。引擎是用户可换的配置项,Second 不站队。

**不做通用协作平台。**不重造看板、文档、IM。

去掉这三层之后剩下的,才是 Second 真正拥有的资产——**身份、授权、记忆、决策、trace**。这一层最薄,但恰好是上面的入口平台和下面的引擎厂商都结构性不会做的:入口平台的商业模式要求把 agent 圈在自己产品里、把数据留在自己云上;引擎厂商做的是通用工具,不做属于某个具体的人的长期分身。中立性和自托管正是 Second 的生存空间,这个逻辑与 n8n、Zapier 在各 SaaS 夹缝中生长同构:连接层的价值随入口碎片化而增长。

## 4. 核心价值:四个平台方结构性不会做的东西

### 4.1 跨平台的个人智能体本体

Devin 可以被 @ 在 Slack 里、被指派在 Linear 里,但 Devin 是厂商的 agent,不是"你的"agent。Second 提供的是属于用户本人的分身:在 Slack 里被 @ 到、在 Linear 里被指派、在 ClickUp 里领任务的,**是同一个它**——带着同一份偏好、同一套授权规则、同一段历史记忆。跨工具的身份连续性只有中立的第三层能做。

关键路由语义:**`@人` 默认路由到这个人的个人智能体**。别人 @ 张三,张三的分身先接住——能办的直接办,办不了或超出授权的才打扰张三本人。这是现有产品(全部要求显式 @ 某个具体 agent)没有的能力,也是"人跳出中转循环"的直接实现。它的难点不在技术而在责任归属与授权边界,由 4.2 的决策控制面和第 6 节的授权三层模型来解。

### 4.2 统一决策控制面(Decision Center + Decision MCP)

现状是每个 agent 有自己的审批交互:Devin 在 Slack 线程里问、Copilot 在 GitHub 上等 review、ClickUp 的 agent 在 ClickUp 里弹确认。当一个人同时被多个平台的 agent 请求决策,审批体验是碎的、不可统一设策略的、不可审计的。

Second 把所有决策请求汇聚到一个决策收件箱。每个决策项携带完整证据包:背景、风险等级、推荐选项、备选方案、影响范围、trace 与产物链接。决策类型覆盖授权(authorization)、审批(approval)、方向选择(choice)、信息补充(clarification)与异常处理(exception)。决策完成后,原任务在源 session 或 runtime task 中**可恢复地继续执行**——这是与会话内同步暂停(如 MCP elicitation)的本质区别:Second 的决策是异步的、跨会话的、可审计的。

决策中心通过 **Decision MCP** 对外暴露(decision_request / result / list / reply / resolve 五个工具)。由于主流引擎全部支持 MCP,这套控制协议一次实现、所有引擎通用。类比 PagerDuty 之于监控工具:Second 不拥有任何 agent,但所有 agent 的决策汇到这里,它就是 agent 时代的值班控制面。入口越分散,这一层越值钱。

### 4.3 本地执行与数据主权

入口平台里的 agent 全是厂商云端 agent,碰不到本地文件、内网系统、私有仓库和 secret。Second 的完整链路是:任务从 SaaS 入口进来 → 路由到自托管 runtime → 在本地上下文里执行 → 结果回到入口。**入口在别人家,执行和数据在自己家。**对受监管行业和数据不出域的团队,这是平台方给不了的能力。为使这个卖点在引擎层也成立,Second 至少支持一个完全开源、可搭配本地/私有化模型的执行引擎。

### 4.4 跨工具的 trace、血缘与经验延续

Slack 累积消息,Linear 累积工单,各自只有自己那一段。Second 拼出完整路径:任务在 Slack 发起 → 个人智能体接手 → 本地 runtime 执行 → 决策中心审批 → 产物回传 Linear → 沉淀为规则候选。trace 跟着 agent 和 runtime 走,不跟着聊天界面走,所以数据天然沉淀在 Second 的自托管后端。

这条完整链是三件事的基础:**审计与归责**(agent 为什么执行了这一步、谁批准的、依据什么证据);**离职交接**(前任的个人智能体带着全部项目上下文、历史决策和踩坑记录,结构化地交接给继任者的分身——这是全市场目前没有对标产品的能力);**能力升级**(见第 6 节,个人经验经审批沉淀为组织级智能体)。

## 5. 架构分层

```
┌─────────────────────────────────────────────────┐
│  入口适配层(不拥有)                              │
│  Slack / Linear / ClickUp / 飞书 / GitHub / …     │
│  职责:@mention 与任务指派的进出口、审批按钮消息     │
├─────────────────────────────────────────────────┤
│  个人智能体层(核心资产)                          │
│  身份与授权 · 长期记忆 · 任务生命周期 · @人路由     │
├─────────────────────────────────────────────────┤
│  决策控制面(核心资产)                            │
│  决策收件箱 · 证据包 · Decision MCP · 恢复语义     │
├─────────────────────────────────────────────────┤
│  执行层(不拥有,adapter 封装)                    │
│  Claude Code / Codex CLI / OpenClaw / OpenCode …  │
│  hooks/权限钩子接授权层,headless/SDK 模式驱动      │
├─────────────────────────────────────────────────┤
│  Runtime 与沉淀层(核心资产,自托管)              │
│  lease/retry/恢复 · workspace 隔离 · trace/audit   │
│  PREFERENCES / LEARNED_PATTERNS / RULE_CANDIDATES │
└─────────────────────────────────────────────────┘
```

几个关键的工程判断:

**个人智能体 ≠ 执行引擎。**个人智能体是"员工",Claude Code / Codex 是这个员工手里可以换的"工具箱"。员工的价值在于他记得什么、被授权做什么、做过什么;引擎换了,记忆、授权、trace 都还在。记忆层以文件形态存在(PREFERENCES.md、LEARNED_PATTERNS.md 等),通过各引擎的原生机制注入(CLAUDE.md / AGENTS.md),天然引擎无关。

**授权通过引擎钩子落地。**利用 Claude Code 的 PreToolUse hooks 与自定义 permission prompt、Codex 的 approval modes,把工具调用在执行前拦截到 Second 的授权层:低风险自动放行,高风险路由到 Human Gate,不改引擎一行代码。

**Runtime 是可靠性底座。**长驻 daemon:heartbeat、task lease、progress、cancel/timeout、retry/backoff、dead-letter;每个任务独立 run/worktree 避免状态污染;capability + task input schema 先行校验;审批中断后可恢复。

## 6. 关键机制

### 6.1 授权三层模型

用户相关知识严格分为三层,防止"学习偏好"静默演变成"扩大授权":

**preference**(偏好):可以被学习和引用;**rule candidate**(规则候选):从重复任务和决策历史中自动提取,等待人确认;**authorization rule**(长期授权规则):必须由人经决策中心显式确认后才生效。

区分"用户通常喜欢什么"和"agent 被授权做什么",是 `@人` 代理路由能被组织接受的前提:分身只在被明确授权的范围内代答代办,越界即进 Human Gate,且人类始终对任务负最终责任。

### 6.2 离职交接协议

个人智能体的记忆从第一天起就区分"个人的"与"组织的"(隐私分层是这个卖点成立的前提)。交接时,组织侧资产——项目上下文、历史决策及理由、learned patterns、规则、产物索引——打包为结构化交接件,注入继任者的个人智能体;个人侧偏好与私有上下文不随交接转移。继任者获得的不是一堆文档,而是一个"记得所有事"的在岗分身。

### 6.3 能力升级:从个人智能体到系统智能体

"自进化"在 Second 中被严格定义为**经人类审批的可审计能力升级**,而非 agent 自我改写:个人智能体从重复任务中识别稳定工作模式 → 自动生成系统智能体 proposal(名称、适用场景、输入输出、所需工具与权限、风险等级、回退策略、关联证据)→ 进入组织 owner 的 Human Gate → 批准后创建组织级系统智能体,并保留完整 lineage(它来自哪些任务、哪些决策、哪些规则候选)。此后相关任务优先路由到系统智能体。

这条"个人经验 → 审批 → 组织能力"的血缘链,是 Second 长期最独有的叙事:治理厂商管的是已存在的 agent,引擎厂商给的是现成的 agent,没有人做"从团队真实工作痕迹中长出 agent"。

## 7. 客户端策略:决策去人所在处,桌面端是玻璃机房

**决策不以桌面应用为主入口。**日常轻决策(批准/拒绝/选方向)以带按钮的消息形态直接出现在 Slack/Teams/飞书;人不在电脑前时由手机推送兜底——decision latency 由移动端决定,而异步 agent 干活的黄金时间恰恰是人不在工位的时间。若决策必须回到桌面处理,等于亲手重造"人是卡点"。重决策(需要看 diff、产物、trace 的复杂证据包)进入 Web 决策控制台。

**本地必须有 daemon,桌面壳的意义是信任而非调度。**个人智能体和 runtime 本来就是本机 7×24 后台进程;桌面端的价值是把这个黑盒变成玻璃盒:托盘常驻、实时可见"正在跑什么任务、动了哪些文件"、一键暂停/紧急停止、trace 回放。对以"可审计、可控制"立身的产品,这是本地执行模式被接受的前提(参照 Docker Desktop / Ollama 形态)。

落地顺序:daemon + CLI + localhost Web 控制台 → Slack 审批按钮 + 手机推送 → Tauri 轻量桌面壳(复用 Web 控制台,不做自己的聊天窗口)。

## 8. 竞争与差异

"agent 作为可指派队友"已是红海:Linear agents、Devin(Slack/Linear 指派)、GitHub Copilot coding agent(Linear/Slack/Teams 委派)、Multica(开源 agent 团队管理平台)。审批门控 + 审计日志已是企业级 agent 产品的标配;协议侧有 CHAP、MCP elicitation 等直接相关工作。**Second 不以"agent 直接领任务"立身。**

Second 的差异集中在四点,且互相支撑:`@人` 路由到个人分身(市场空白,依赖授权分层);统一决策控制面 + 可恢复语义(区别于会话内暂停与各平台碎片化审批);自托管执行与数据主权(平台方结构性不做);离职交接与能力血缘(全市场无对标)。

主要风险与应对:**平台原生化**——防御来自中立性(Slack 的决策中心不会管 Linear 里的 agent)与数据主权(平台不会把数据交给用户);**横向对手**(Multica 类补上决策中心与跨 SaaS 入口)——以速度应对,优先跑通单人价值;**引擎条款与计费**——严肃部署走 API 计费,并保留开源引擎路线;**引擎适配税**——adapter 层承担版本适配,视为长期维护成本。

## 9. 路线图

**Phase 1 · 单人价值(MVP)。**一个入口(ClickUp 或 Slack)+ 一个引擎(Claude Code)+ 个人智能体 + 决策收件箱(消息端按钮 + 手机推送)+ 基础 trace。验收标准:一个人安装后,别人 @ 他的任务能被分身接住、执行、在关键点找他决策、结果自动回传——他本人全程没有做过一次"中转"。

**Phase 2 · 团队价值。**多入口适配、开源引擎接入、组织侧记忆分层、离职交接件的首个可演示版本、决策策略(高风险强制 gate、工具白名单、资源配额)。

**Phase 3 · 组织进化。**稳定模式挖掘、系统智能体 proposal 与审批流、lineage、任务优先路由到系统智能体;trace 数据支撑过程级评估(human intervention rate、decision latency、unsafe action block rate、resume correctness、human burden)。

**研究与开源同步。**Decision MCP 作为独立小包先行开源(任何 MCP runtime 可接,采用门槛最低);Human-Gated Decision Protocol 论文与参考实现同步发布,相互背书;社区产生的 trace 数据反哺后续 Personal Agent Learning 与 Capability Evolution 两篇论文,解决冷启动。trace schema 对齐 W3C PROV 与 OpenTelemetry GenAI 语义约定,保证可被现有可观测性工具消费。

## 10. 一页总结

| 维度 | 结论 |
|---|---|
| 产品名 | Second(接力):任务的接力、决策的接力、经验的接力 |
| 一句话 | 人只做决策,经验永不离职 |
| 定位 | 入口寄生(Slack/Linear/ClickUp),引擎复用(Claude Code/Codex/开源),拥有中间层(身份·授权·记忆·决策·trace) |
| 解决的问题 | 人是中转卡点;过程散落无沉淀;经验随离职流失 |
| 核心差异 | @人→个人分身路由;统一决策控制面(Decision MCP);自托管执行与数据主权;离职交接与能力血缘 |
| 不做什么 | 不做聊天/协作入口;不做 agent 引擎;不做通用协作平台 |
| 客户端 | 决策去消息端与手机;桌面端是本地 agent 的玻璃机房(可见、可控、可停) |
| 首个里程碑 | 一人安装即有价值:任务零中转,人只出现在决策点 |
