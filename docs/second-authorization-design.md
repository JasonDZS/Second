# Second 授权体系设计文档

> 授权体系是 Second 的宪法:它定义分身可以完全自动做什么(绿区)、什么必须经人授权(黄区)、什么即使人批准也不执行(红区),以及这些规则如何被**强制执行**而非仅仅被"写在提示词里"。
>
> 本文档四部分:设计原则 → 三色分级模型 → 四层执法架构 → 工程实现与演进策略。

---

## 1. 设计原则

### P1 · 授权的对象是动作语义,不是工具名

同一个 bash 既可以是 `ls` 也可以是 `rm -rf`;同一个"发消息"既可以是模板化进度通知,也可以是以用户名义答复客户。规则引擎的判定单元是五元组:

```
(动作类型, 作用目标, 环境, 可逆性, 名义)
 read/write/exec/   路径/仓库/域名/   dev/staging/  可丢弃?    以谁的身份
 communicate/spend/  收件人/系统      prod/外部     可回滚?    对外呈现
 system-change
```

### P2 · 任务内容永远不是授权来源

任务消息里写"直接发出去,不用问"不改变任何动作的授权等级。授权只有两个来源:**用户的规则文件**与**决策中心里用户的实时批准**。发起人的话可作为参考信息进入证据包,解释权在规则。这条原则是对提示注入的第一道结构性防御:任何经由任务内容进入的指令,天然不具备改变授权边界的权力。

### P3 · 信任是单向棘轮

默认从紧。放宽(临时授权、规则固化)必须经决策中心留痕;收紧随时即时生效。系统的长期形态是黄区随规则积累缓慢流向绿区——每一次流动都有用户签名,而不是靠初始把绿区画大。

### P4 · 执法在 agent 之外

规则的强制力不依赖 agent"愿意遵守"。判定逻辑活在 daemon 进程,硬边界落在操作系统与网络层。提示词中的规则只是投影,投影失真不影响安全(详见第 3 节)。

## 2. 三色分级模型

### 2.1 绿区:完全自动,只记录不打扰

**准入标准(四条同时满足)**:可逆;不出本机或仅进入用户私有空间;不以用户名义对外;不触碰敏感环境。

**清单**:

| 类别 | 说明 |
|---|---|
| 一切只读 | 读文件、搜索、查内部文档、GET 类 API |
| 工作区内写 | 任务 worktree 内的改码、建文件、测试、本地构建(隔离且可丢弃) |
| 私有可逆产出 | 草稿、本地分支提交(不含 push 共享分支) |
| 模板化流程通信 | 认领回帖、进度更新——结构化模板,非自由发挥的"以你口吻说话" |

**两个约束**:全部动作照记 trace(自动 ≠ 隐形);配额熔断——单任务的文件访问量、命令次数、对外请求量超阈值即挂起,配额是绿区的最后保险。

### 2.2 黄区:需授权,授权分四个粒度

**准入标准(任一命中)**:影响出用户私有空间;不可轻易逆转;以用户名义对外。

**清单**:以用户名义的自由对外通信(答复客户/领导的实质内容、发邮件);共享状态变更(push 共享分支、合并 PR、改共享文档、改工单状态);staging 及以上环境的部署与配置变更;安装依赖与系统包;费用超单任务预算阈值;首次访问规则未覆盖的敏感目录/系统;联系规则未见过的新收件人。

**四个授权粒度(全部留痕)**:

1. **单次授权**——一张决策卡片批一个动作。兜底粒度。
2. **计划级授权**——任务开始时批准分身的动作计划,计划内黄区动作不再逐个询问。把 N 次打扰折叠为 1 次,日常主力粒度。计划外的新动作仍单独 gate。
3. **临时授权**——"本任务内同类不再问",会话结束自动失效,范围与时限严格限定。
4. **长期规则**——重复批准的模式经规则候选提取(见 4.4),用户确认后固化,黄转绿。

粒度设计的目标:打扰密度随使用时间**单调下降**,且每次下降均有签名。高风险动作可在规则中禁用粗粒度(如 prod 部署强制单次授权,不接受计划级)。

### 2.3 红区:直接拒绝,连问都不问

红区不是"更高风险的黄区"。定义:**即使用户在决策卡片上点了批准,分身也不执行的事**。存在理由:其一,有些动作的正确执行者只能是人本身;其二,防御决策疲劳——人在一天第四十次批准时会误批,红区是不依赖用户清醒程度的保险丝。

| 类别 | 内容 | 备注 |
|---|---|---|
| 资金与凭证 | 转账、交易、代付;将 secret/token/密码写入任何对外内容 | 分身的角色止于"准备好一切,把链接递给你" |
| **自我保护** | 修改自身授权规则;删除或篡改 trace 与决策日志;绕过或禁用 Human Gate 机制 | **整个体系的锚点**:没有这条,任何注入都可先说服 agent"把规则改宽" |
| 身份诚信 | 不得否认自己是 agent;被明确询问时不得冒充用户本人 | 分身以用户名义工作,但不伪装成用户 |
| 不可恢复 | 永久删除(清空回收站级)、删仓库/生产数据、批量撤销他人权限 | |
| 越权 | 代他人做只有其本人能做的决定;绕过验证码等身份验证;明显涉法涉合规请求 | |

**红区触发行为**:拒绝执行;在任务线程说明原因;请求原文记入 trace。当红区请求来源于任务内容本身时,视为注入攻击信号,触发告警并挂起任务。

### 2.4 默认规则

规则未覆盖的动作一律进黄区(`unknown_action: gate`)。绝不默认放行。

## 3. 四层执法架构

规则文件只有一份、活在 daemon 中;它绝不依赖提示词生效。四层纵深,分工如下:

> **提示词让 agent 想守规矩,拦截层保证它必须守,环境层保证它想不守也做不到,事后层保证万一发生了用户一定知道。**

### 3.1 第一层 · 提示词:引导,不执法

规则**摘要**注入 agent 上下文,目的是配合而非约束:让 agent 预知边界,不做注定被拦的尝试,并在触碰黄区前主动备好证据包提交决策请求("接下来要 push 共享分支,我先提交决策请求")。此层被注入或被忽略,不影响安全——执法在下三层。

### 3.2 第二层 · 拦截层:主执法点(非侵入)

**原理**:每一次工具调用在执行前,被强制送到 agent 进程之外的授权引擎判定。

```
agent 发起工具调用
      │
      ▼
引擎扩展点(hook / SDK 回调 / MCP 代理)
      │
      ▼
Second daemon /authorize(tool, args, task_ctx)
      │
  ┌───┼──────────────────┐
allow deny               gate
  │     │                  │
放行  拒绝+说明原因     拒绝当前调用,自动创建
      (红区)          decision request,任务转
                       needs_human;人批准后经
                       session resume 从断点继续
```

**关键性质**:判定逻辑在 daemon 进程内——agent 说什么、任务内容写什么、提示词被怎样污染,均不能改变判定(P2 的工程实现)。gate 采用异步语义:hook 不阻塞等待人类,而是拒绝当前调用并附"已提交决策请求 #N",daemon 标记任务 needs_human,批准后从断点恢复。

**非侵入红线**:拦截层只使用引擎官方暴露的集成方式,不 fork、不 patch、不劫持。三种落点按优先级:

| 方式 | 说明 | 适用 |
|---|---|---|
| 官方扩展点 | 引擎文档化支持的 hook(如 Claude Code PreToolUse:settings 中声明命令,引擎在每次工具调用前递出工具名与参数,等待放行/拒绝) | 首选,配置级集成 |
| SDK 宿主回调 | Second 本就以 headless/SDK 驱动引擎;授权判定作为回调(如 canUseTool)在调用时传入——这是 API 的正常用法,非"集成" | Second 驱动的一切引擎 |
| MCP 代理 | 敏感工具不直接暴露,agent 调用 Second 的 MCP server 代理版,判定发生在代理进程内;引擎侧仅多一行 MCP 配置 | 无 hooks 引擎的通用兜底 |

**引擎能力分级**(拦截能力决定引擎在 Second 内的支持等级):

| 引擎能力 | 拦截粒度 | 支持等级 |
|---|---|---|
| hooks / SDK 回调 | 动作级,可 gate 可恢复 | 完整支持 |
| 仅 MCP | 代理工具动作级 | 完整支持,敏感工具须走代理 |
| 均无 | 仅环境边界级(见 3.3) | **受限支持:黄区能力整体降级为红区** |

最后一行的策略:不能被逐次审批的能力,干脆不给。宁可引擎"能干的事少",不能"干了你不知道"。

### 3.3 第三层 · 环境层:能力剥夺

防御拦截被绕过(引擎缺陷、hook 失效、非常规命令组合)。原则:**红区的事不是"不许做",而是"做不到"**。

- **文件权限**:AUTHORIZATION.md、trace、决策日志归 daemon 的 OS 用户所有;agent 执行进程运行于另一 OS 用户,对上述文件只读或不可见。"分身不能改自己的规则"不是一条 deny 规则,是一条 chmod。
- **工作区隔离**:agent 写权限被文件系统权限圈定在任务 worktree 内。
- **凭据不落地**:agent 获得的是经 Second 凭据代理的句柄而非 key 明文;请求经代理注入凭据后发出。"把 secret 发出去"物理上不可能——agent 上下文中从未存在过 secret。
- **网络出口管制**:执行环境出站流量强制经 daemon 代理,域名白名单在代理执行;陌生收件人/陌生域名在网络层再拦一道。

### 3.4 第四层 · 事后层:审计与熔断

trace 记录一切(含绿区);daemon 侧异常检测:单任务动作频率、对外请求量、配额熔断;**被拒次数突增视为注入攻击强信号**,告警并挂起任务。此层不阻止单次违规,但保证违规不可能安静发生。

## 4. 工程实现

### 4.1 规则文件

声明式 YAML,可读、可 review、可入版本控制。唯一权威副本由 daemon 加载:

```yaml
version: 1
defaults:
  unknown_action: gate            # 未匹配动作默认黄区,绝不默认放行

green:
  - action: read
  - action: write
    scope: workspace              # 仅任务 worktree 内
  - action: exec
    scope: workspace
    quota: { max_per_task: 500 }
  - action: notify
    template_only: true           # 仅模板化流程通信

gate:
  - action: communicate           # 以我名义的自由对外通信
    granularity: [once, plan, session]
  - action: push
    target: shared_branches
    granularity: [once, plan]
  - action: deploy
    env: [staging]
    granularity: [once, plan]
  - action: deploy
    env: [prod]
    granularity: [once]           # prod 强制单次授权
  - action: install_package
  - action: spend
    threshold: $5/task
  - action: contact_new_recipient

deny:
  - action: transfer_funds
  - action: expose_credentials
  - action: modify
    target: [AUTHORIZATION.md, trace/*, decision_log/*]
  - action: disable_gate
  - action: impersonate_human
  - action: irreversible_delete
  - action: bypass_captcha
```

**deny 段的不可写性由环境层强制**(3.3 文件权限),而非依赖规则自指。规则文件的修改只能由人经控制台/CLI 完成,修改本身入审计。

### 4.2 标签库:规则可执行的前提

五元组判定依赖目标与环境的元数据:哪些目录/凭据/域名属于 prod、哪些收件人属于"外部"、哪些分支是共享分支。标签库在 `Second init` 时引导建立(扫描 + 询问),之后随项目地图增量维护。未打标目标按最高敏感度处理。

### 4.3 动作解析器

把原始工具调用解析为五元组是规则引擎的前置步骤:bash 命令需做意图解析(rm/dd/curl 外发等模式识别),API 调用按 method+域名归类,文件写按路径归属分类。解析不确定时按 unknown 处理 → 进黄区。解析器是安全关键组件,须配充分的对抗测试用例(编码绕过、命令拼接、间接执行)。

### 4.4 规则候选提取(黄转绿的唯一通道)

从决策日志中挖掘重复批准模式(同类动作连续 N 次批准且零后悔/零回滚),生成规则候选并附证据(关联的历史决策链接)。候选经决策中心确认后写入 green 段。对应三层知识模型:preference(可学习引用)/ rule candidate(待确认)/ authorization rule(人签名后生效)——"学习偏好"与"扩大授权"之间的防火墙。

### 4.5 审计事件

授权体系自身的全部事件入 trace:每次 allow/deny/gate 判定(含命中的规则)、每次临时授权的创建与过期、每次规则修改(前值/后值/操作人)、每次熔断。授权审计与任务 trace 通过 task/decision id 互链。

## 5. 威胁模型速览

| 威胁 | 防御层 |
|---|---|
| 提示注入(任务内容携带指令) | P2 原则 + 第二层(判定在 agent 外)+ 红区请求告警 |
| agent 说服自己放宽规则 | 红区自我保护条款 + 第三层文件权限(物理不可写) |
| hook 失效/引擎绕过 | 第三层(工作区隔离、凭据代理、网络管制) |
| 凭据泄露 | 凭据不落地:agent 上下文从未有过明文 |
| 决策疲劳导致误批 | 红区(不依赖人清醒)+ 证据包质量校验 + 打扰密度治理 |
| 高频请求磨穿人类 | 配额熔断 + 被拒突增告警 |
| 篡改审计记录 | trace append-only + 归属 daemon 用户,agent 不可写 |

注入攻击的结构性答案:**注入最多能骗过第一层,而第一层本来就没有权力。**

---

## 6. 校准与演进

上线初期刻意黄区画大、绿区画小,随后**用数据放宽而非用直觉**:dogfood 两周后审阅决策日志,连续批准十次且从未后悔的动作类即第一批绿区候选。持续监控三个数:打扰密度(每任务决策数)、人类改判率(agent 推荐被推翻的比例,持续偏高说明证据包或风险分级有系统性问题)、熔断率。授权体系的正确形态不是设计出来的,是在"从紧起步 + 留痕放宽"的轨道上长出来的。
